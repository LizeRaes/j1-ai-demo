# Agents Guide: MediFlow Ticketing System

This document provides essential information for AI agents and developers working with the MediFlow Ticketing System codebase.

## System Overview

**Purpose**: Support ticketing system for MediFlow (a digital healthcare logistics application). This system is used by customer service and engineering teams to handle user complaints, questions, and bug reports.

**Architecture**: 
- Backend: Quarkus 3.30.8 (Java 17) with Hibernate ORM + Panache
- Database: MySQL 8 (Oracle-migratable design)
- Frontend: Vanilla HTML/CSS/JavaScript (no framework)
- API: RESTEasy Reactive (JAX-RS)

## Key Design Choices

### 1. Database Portability (Oracle Migration Ready)

**Why**: System must be able to migrate from MySQL to Oracle in production.

**Implementation**:
- Uses standard SQL types: `VARCHAR`, `CLOB`, `TIMESTAMP` (not MySQL-specific)
- Avoids MySQL `JSON` type - uses `CLOB` for JSON blobs
- Boolean fields stored as `TINYINT(1)` in MySQL (Hibernate maps to Boolean)
- Uses `GenerationType.IDENTITY` for primary keys (works on both)
- Entity definitions use standard JPA annotations (Oracle-compatible)

**Files**:
- `src/main/java/com/mediflow/ticketing/domain/model/*.java` - Entity definitions (schema is auto-generated by Hibernate)

**To change database**:
- Update `src/main/resources/application.properties`:
  - `quarkus.datasource.db-kind=mysql` → `oracle`
  - `quarkus.datasource.jdbc.url` → Oracle connection string
  - `quarkus.datasource.username` and `password`

### 2. Layered Architecture (Separation of Concerns)

**Structure**:
```
api/          → REST endpoints (no business logic)
service/      → Business logic (transactions, orchestration)
persistence/  → Data access (Panache repositories)
domain/        → Entities and enums
dto/          → Data transfer objects (API contracts)
mapper/       → Entity ↔ DTO conversion
```

**Rule**: API layer (`api/`) has NO business logic. All business logic is in `service/` layer.

**Key Files**:
- `src/main/java/com/mediflow/ticketing/api/*Resource.java` - REST endpoints
- `src/main/java/com/mediflow/ticketing/service/*Service.java` - Business logic
- `src/main/java/com/mediflow/ticketing/persistence/*Repository.java` - Data access

### 3. Event-Driven Logging

**Why**: Real-time visibility into system activity for support teams.

**Implementation**:
- All significant actions write to `event_log` table
- Events are color-coded in UI by type and severity
- Frontend polls `/api/events/recent` every 1.5 seconds
- Events include: ticket creation, status changes, comments, dispatch actions, errors

**Key Files**:
- `src/main/java/com/mediflow/ticketing/service/EventService.java` - Event logging
- `src/main/java/com/mediflow/ticketing/domain/model/EventLog.java` - Event entity
- `src/main/resources/META-INF/resources/js/ui/renderLogs.js` - Frontend polling

**Event Types** (see `EventType` enum):
- `INCOMING_REQUEST_RECEIVED` - Green
- `TICKET_CREATED` - Green
- `SYSTEM_STEP` - Yellow
- `RETURNED_TO_DISPATCH` - Blue
- `NOTIFICATION_SENT` - Red (future)
- `ERROR_OCCURRED` - Orange

### 4. Three Ticket Creation Paths

**Path A: Incoming Request → AI Triage → Ticket**
1. User complaint arrives via `POST /api/incoming-requests` or `POST /api/intake/incoming-request`
2. Request is immediately processed by AI triage worker (synchronous, will become async later)
3. On success: Creates ticket with `source=AI`, `status=FROM_AI`, `rollbackAllowed=true`
4. On failure: Request marked as `AI_TRIAGE_FAILED` and appears in "Dispatcher Inbox"
5. Dispatcher can manually dispatch failed requests (same as Path A manual flow)

**Path B: Manual Ticket Creation**
- Direct creation via `POST /api/tickets/manual`
- Used by admins/demos
- Default: `source=MANUAL`, `status=FROM_DISPATCH`

**Path C: AI Ticket Creation (Direct)**
- Via `POST /api/tickets/from-ai`
- Creates ticket with `source=AI`, `status=FROM_AI`, `rollbackAllowed=true`
- UI shows "Accept" or "Reject & Return to Dispatch" buttons

**Key Files**:
- `src/main/java/com/mediflow/ticketing/service/TicketService.java` - All creation methods
- `src/main/java/com/mediflow/ticketing/service/DispatchService.java` - Dispatch flow
- `src/main/java/com/mediflow/ticketing/api/TicketResource.java` - REST endpoints

## Database Schema

### Core Tables

#### `incoming_requests`
- Stores user complaints before they become tickets
- Status: `NEW` → `DISPATCHED` → `CONVERTED_TO_TICKET`
- Fields: `user_id`, `channel`, `raw_text` (CLOB), `status`, timestamps

#### `tickets`
- Main ticket records
- Fields:
  - `original_request` (CLOB) - Immutable user complaint text
  - `ticket_type` - Enum: BILLING_REFUND, BILLING_OTHER, SCHEDULING_CANCELLATION, etc. (see TicketType enum)
  - `status` - Enum: FROM_DISPATCH, FROM_AI, TRIAGED, IN_PROGRESS, etc.
  - `source` - Enum: MANUAL, AI, API
  - `assigned_team` - String (team name)
  - `assigned_to` - String (user ID, nullable)
  - `urgency_flag` - Boolean
  - `urgency_score` - Double (nullable)
  - `ai_confidence` - Double (nullable, for AI tickets)
  - `rollback_allowed` - Boolean (true for AI tickets)
  - `ai_payload_json` - CLOB (nullable, future AI data)
  - `incoming_request_id` - FK to incoming_requests (nullable)

#### `ticket_comments`
- Comments on tickets
- Fields: `ticket_id` (FK), `author_id`, `body` (CLOB), `created_at`

#### `event_log`
- System events for real-time logging
- Fields:
  - `ticket_id` (FK, nullable)
  - `incoming_request_id` (FK, nullable)
  - `event_type` - Enum
  - `severity` - Enum: INFO, WARNING, ERROR
  - `source` - String (e.g., "ticketing-api", "dispatch-ui")
  - `message` - String (512 chars)
  - `payload_json` - CLOB (nullable)
  - `created_at` - Timestamp

### Database Schema
**Management**: Hibernate auto-updates schema on startup (`quarkus.hibernate-orm.database.generation=update`)

- Tables are created/updated automatically by Hibernate based on JPA entities
- Schema is defined in entity classes in `src/main/java/com/mediflow/ticketing/domain/model/`

**Location**: `src/main/java/com/mediflow/ticketing/domain/enums/TicketType.java`

**Values**: See "Event-Driven Logging" section above.

## Transaction Management

**Pattern**: All service methods that modify data are `@Transactional`.

**Key Files**:
- All `*Service.java` files use `@Transactional` on write methods
- Read methods are NOT transactional (read-only)

**Example**:
```java
@Transactional
public TicketDto createTicketManual(CreateTicketManualDto dto) {
    // Creates ticket and logs event in same transaction
}
```

## Error Handling

**Current State**: Basic error handling via HTTP status codes.

**Pattern**:
- REST resources return `Response.status(404)` for not found
- Services throw `IllegalArgumentException` for bad input
- Frontend shows alerts for errors

**To improve**: Add exception mappers in `api/` layer.

## Testing Considerations

**No tests included** (per requirements), but structure supports:
- Service layer is easily testable (inject mocks)
- Repositories use Panache (testable with Testcontainers)
- REST endpoints can be tested with `@QuarkusTest`

## Performance Considerations

### Database
- Indexes on hot query paths (see schema)
- Connection pool configured (2-20 connections)
- Short transactions (write and commit quickly)

### Frontend
- Event polling: 1.5 second interval (configurable)
- Event limit: 200 events max
- Incremental event loading (only new events after initial load)

## Common Tasks

### Add a New Ticket Status
1. Add enum value to `TicketStatus.java`
2. Update UI rendering in `renderTicketDetail.js` (status dropdown)
3. Hibernate will auto-update schema (enum stored as VARCHAR)

### Add a New Event Type
1. Add enum value to `EventType.java`
2. Update color mapping in `js/util/format.js` (`getEventColorClass()`)
3. Add event logging call in relevant service method

### Change Event Polling Frequency
**File**: `src/main/resources/META-INF/resources/js/ui/renderLogs.js`
```javascript
pollInterval = setInterval(loadEvents, 1500); // Change 1500 to desired milliseconds
```

### Add a New Team
1. Add enum value to `Team.java`
2. Update UI dropdowns in:
   - `renderDispatcherInbox.js` (dispatch form)
   - `renderCreateTicket.js` (create form)

### Modify Ticket Creation Flow
**File**: `src/main/java/com/mediflow/ticketing/service/TicketService.java`
- Methods: `createTicketFromDispatch()`, `createTicketManual()`, `createTicketFromAi()`
- Each method creates ticket and logs event

### Change Database Connection
**File**: `src/main/resources/application.properties`
- Update `quarkus.datasource.*` properties
- For Oracle: change `db-kind=oracle` and update JDBC URL

## Important Notes

1. **No Business Logic in API Layer**: All business logic must be in `service/` layer
2. **Always Log Events**: Use `EventService.logEvent()` for significant actions
3. **Oracle Compatibility**: Keep SQL and types Oracle-compatible
4. **Short Transactions**: Keep database transactions short
5. **Event Polling**: Frontend polls events, not WebSockets (for stability)
6. **Demo Data**: 5 incoming requests seeded in `V2__seed_demo_data.sql`
